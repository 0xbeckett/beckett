Why it feels broken

One brain, two jobs, zero separation

Both symptoms you feel trace to a single design choice: today's Beckett is one long-lived claude -p agent that is simultaneously your conversation partner and the work orchestrator. Discord chat, planning, worker spawning, smoke-alarm signals, and log digests all arrive on one stdin, into one session.

Symptom 1 · Chat gets lobotomized

  • Your conversation shares a context window with worker telemetry, signals, and DAG state.
  • Auto-compaction is what keeps that session alive — so the conversational thread is the first thing summarized away.
  • The agent constantly task-switches between "talk to Jason" and "babysit worker 3," so its voice and focus drift.

Symptom 2 · Parallelization fails

  • One reasoning loop is the serialization point — it must plan, dispatch, observe, nudge, and integrate, all in sequence.
  • While it's deep in a reply to you, nobody is advancing the workers (and vice-versa).
  • "Spawn N workers" is just a tool call from a brain that's already overloaded — there's no substrate that actually runs them concurrently on its behalf.

The v3 thesis: separate conversation from orchestration from execution, and give parallelism a real home instead of asking one overloaded loop to fake it. The four options below make that cut in increasingly aggressive ways.

Baseline

Today — v2 single parent SHIPPING

A thin bun shell pumps every input onto one parent agent's stdin. That parent is the chat partner, the planner, and the supervisor. Workers are tool calls it makes and then has to watch — inside the same context it's talking to you in.

What hurts

  • Context conflation — chat + orchestration share one window; compaction eats the conversation.
  • Serial brain — chatting blocks orchestrating; orchestrating blocks chatting.
  • Shallow parallelism — the parent can't truly mind N workers while also being present for you.

What's worth keeping

  • Reasoning-driven control — no hand-coded state machine; the model decides.
  • Worktree + hooks + digests — the worker isolation & telemetry layer is solid.
  • Dynamic effort — inline / one-worker / heavy-path is the right instinct.
Option A · smallest change

Split Brain FRONT / BACK OF HOUSE

Keep two persistent agents instead of one. A Concierge owns Discord and your conversation — always warm, never sees a worker log. When real work appears, it files a structured ticket to a separate Orchestrator and steps back. They share memory and a task ledger, but never share a context window.

✓ Fully fixes the lobotomy · ◐ helps parallelism

Why it helps

  • Chat stays pristine — the Concierge's window never holds a single worker log.
  • Two loops run at once — you can chat while the Orchestrator drives work.
  • Smallest leap from today: it's the current parent, cloned and given one job each.

Limits / cost

  • Parallelism is still one orchestrator loop — better, but not unbounded.
  • Need a clean handoff contract (ticket in, result out) and a sync'd ledger.
  • Two long-lived sessions to supervise & resume instead of one.
Option B · the sweet spot

Ephemeral Conductors ONE LEAD PER TASK

No standing brain for work at all. A thin, cheap Router handles every mention: trivial → it replies inline; real task → it spawns a fresh, dedicated Conductor that lives only for that task — plans it, fans out its own workers, integrates, delivers, and dies. Two tasks = two independent conductors, born clean, running side by side.

✓ Fixes both — clean context per task AND real parallelism

Why it helps

  • Every task gets a clean head — no accumulating cruft, no cross-task confusion, ever.
  • Parallelism is the default, not a heroic act — N tasks literally are N processes.
  • Chat is never blocked: the Router is light and hands off instantly.
  • Crash blast radius = one task. Resume = re-spawn that conductor from its ledger.

Limits / cost

  • No cross-task memory in-context — shared awareness must live on disk (KG + ledger) and be re-read.
  • Spawn cost & cold-start per task; the Router must classify well or you over/under-spend.
  • "Who owns this repo right now?" needs a coordination layer so two conductors don't collide.
Option C · most scalable

Blackboard Swarm EVENT-DRIVEN · NO CENTRAL LOOP

Dissolve the brain entirely. A shared blackboard — a task board + event bus — is the spine. Stateless roles (planner, worker, reviewer, deliverer, chat) wake when an event matches them, do their bit, write back, and sleep. The "orchestrator" shrinks to a scheduler/policy, not a conversationalist.

✓ Max parallelism & resilience · ✗ hardest to build & to give one voice

Why it helps

  • Unbounded parallelism — any number of roles act off the board at once; the scheduler just caps it.
  • Most resilient — every role is stateless & replaceable; the board is the durable truth.
  • Chat is just another subscriber — never tangled with execution state.

Limits / cost

  • Most engineering — you build the bus, the board schema, claim/lease logic, dedup.
  • Hardest to give a coherent voice — "Beckett" becomes emergent, not a single speaker.
  • Reasoning is fragmented across many short-lived calls — global judgment is harder.
Option D · familiar org-chart

Manager Tree HIERARCHICAL DELEGATION

Keep one thin top Coordinator, but it only delegates — never does the work or watches leaf workers directly. It spins up mid-level Leads, one per sub-goal, and each Lead manages its own worker pool. A dedicated Chat branch keeps your conversation isolated from the whole tree.

◐ Scales parallelism by breadth · adds layers & latency

Why it helps

  • Parallelism scales with breadth — add Leads to add concurrent fronts; each has a bounded context.
  • One coherent owner at the top, so "Beckett" still has a single decision-maker & voice.
  • Chat branch is isolated, so conversation stays clean.

Limits / cost

  • More layers = more latency & more places for intent to get garbled going down/up.
  • The Coordinator can still bottleneck if it micromanages instead of truly delegating.
  • Overkill for the medium tasks that are most of the volume.

My read

Where I'd point v3

The two pains have one cause, so the fix is one cut: stop letting conversation and orchestration share a context window. Every option above makes that cut — they differ only in how far.

If I had to choose: start at A, design toward B. Splitting the parent into a Concierge + Orchestrator is a small, low-risk change that immediately stops the chat from being lobotomized — and it's mostly reusing today's parent twice. But A still has one orchestration loop, so it only softens the parallelism problem. B (Ephemeral Conductors) is where real parallelism lives: a clean head per task is exactly what kills both the cross-task confusion and the serial bottleneck, and it reuses your strong worker/worktree/digest layer underneath each conductor. C is the endgame if Beckett ever needs to scale past a handful of concurrent tasks, but it's a much bigger build and trades away the single coherent voice you've cultivated. D fits specific big jobs but is overkill for the medium-task volume.
