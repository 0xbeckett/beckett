# OPS-125 — Blip-proof workers: survive a daemon restart without losing in-flight work

## Problem

Every live worker is a **child process of the daemon**. When the daemon restarts — a deploy, a
manual `systemctl restart`, or a genuine crash — the worker dies with it. Over ~25 restarts in 3.5
days that is a lot of murdered work. The goal is not to stop the restarts (most are us); it is to
**survive** them: an interrupted worker should resume from where it was, not restart its stage from
zero.

This ticket is the **worker side only**. The concierge's own conversation-context loss is a
separate, later ticket and is explicitly out of scope here.

## Decision: clean RESUME, not live REATTACH

We ship **clean RESUME** — on boot, respawn a *fresh* worker process that continues the interrupted
harness session from a durable checkpoint — and deliberately do **not** do live REATTACH (adopting
the orphaned OS process).

### Why RESUME won

- **The expensive thing to preserve is context, and the harness already persists it.** claude and
  pi own their session identity from `t=0` via a pre-minted session UUID (`spawn.ts`). Relaunching
  in resume mode against that id restores the whole transcript — the worker's accumulated reasoning
  — for the price of one process spawn. That is the bulk of what a restart used to throw away, and
  it needs no process adoption at all.
- **The dispatcher already has the restart-memory to build on.** Issue #20 landed the ledger
  (`DispatcherRuntimeState.liveWorkers`), the boot orphan-sweep + ghost-WIP commit
  (`recoverFromCrash`), and the session-resume hint plumbed into `spawnWorker({ resumeSessionId })`.
  RESUME extends that machinery; REATTACH would replace it.
- **REATTACH is genuinely gnarlier for little marginal gain.** Re-adopting a live OS process means
  detaching workers (`setsid`) so they outlive the daemon, tracking and re-opening their stdio/IPC
  across the restart, reparenting PIDs, and reconciling a process that kept *mutating the worktree*
  while the daemon was down (no watchdog, no event capture, no stall detection). All of that to save
  a single process respawn — because the context is already recoverable via the session id. The
  risk/complexity is not worth it for v1.
- **RESUME degrades safely.** A stale session file or harness drift makes resume fail; `doSpawn`
  already falls back to a fresh worker rather than stranding the ticket. REATTACH has no equally
  clean fallback — a half-adopted process is worse than a dead one.

REATTACH remains a possible v2 if we ever find work that resume genuinely can't reconstruct (e.g. a
harness with no durable session). It is filed as a follow-up, not wedged in here.

## What this ticket adds on top of issue #20

Issue #20 already resumes the *session* and commits *on-disk* ghost WIP at boot. The gap it left is
the **loss window**: if the daemon is hard-killed (SIGKILL / OOM / power) the graceful shutdown
drain never runs, so the only WIP that survives is whatever happened to be on disk at crash time,
committed once at the next boot. OPS-125 bounds that window explicitly.

### 1. Periodic worktree checkpoint (`supervise.worker_checkpoint_s`, default 120s)

`Dispatcher.startCheckpointLoop()` (armed at boot, right after `recoverFromCrash`) runs
`checkpointLiveWorkers()` on a cadence. Each pass commits every live worker's worktree as a WIP
checkpoint using the **same** `commitWorktree` the boot sweep and the finish path already use, and
records `lastCheckpointAt` / `lastCheckpointSha` on the existing ledger entry.

- **Commit, not a side snapshot.** A committed checkpoint is picked up by the exact paths that
  already handle committed WIP — the resume worker sees it in `git log`, the reviewer diffs
  `baseSha..HEAD` over it, publish pushes it. A separate snapshot ref would be a parallel recovery
  path to build and maintain (and would violate "extend, don't duplicate"). The review diff base is
  unchanged (captured once at first implement), so checkpoints are just in-progress commits, exactly
  like a retry-from-committed-WIP.
- **Best-effort and isolated.** Per-worker `try/catch`; a git failure (e.g. an `index.lock` race
  with the worker's own commit) is logged and skipped, never fatal, and one worker's failure never
  blocks the others. A worker that has already produced its terminal result is skipped so the
  checkpoint never races its `onDone` commit. Overlapping ticks are coalesced (`checkpointInFlight`).
- **Side-effect-free beyond the worktree.** It never touches Plane, the advance-outbox, or the
  publish-outbox — so it cannot affect finish-path idempotency (see below).

Result: a hard crash loses at most one checkpoint window of in-flight edits, not the whole session.

### 2. Enriched resume brief (`resume-brief.ts`)

The continuation instruction handed to a resumed worker now names the four durable anchors OPS-125
requires: **ticket**, **stage**, the review-diff **base SHA**, and a pointer to its **accumulated
context** — the restored transcript plus the committed checkpoint trail (`git log --oneline`), which
survives even if the transcript resume degrades to a fresh session. Extracted into its own module so
it is unit-testable in isolation (the dispatcher tests mock `./spawn.ts` process-globally).

### 3. Observability

`beckett status` now reports `lastCheckpointAgeSecs` per live worker — the current crash loss floor
at a glance.

## Idempotency (the load-bearing constraint)

This is the core dispatcher finish path; a bad merge breaks Beckett. The checkpoint work touches
**only** worktree git state and the in-memory/on-disk ledger. It calls no Plane API and neither
outbox. The finish path is unchanged:

- **advance-outbox** still guards Plane state moves (`applyAdvance` re-reads the ticket and skips a
  terminal-move-that-a-human-won); a restart mid-advance replays the queued row without
  double-advancing.
- **publish-outbox** still owns the worktree for a queued publish and is idempotent
  (`ensurePublished` is safe to rerun); a restart mid-publish replays without double-publishing.
  Publish-held tickets have **no live worker**, so `checkpointLiveWorkers` never touches them.

## Restart-mid-stage correctness

The ledger records the worker's `stage`. On boot `recoverFromCrash` arms a resume hint keyed by
`(ticket, stage)`; the re-emitted `state_changed` (`in_progress` → implement, `in_review` → review)
spawns the matching stage and `doSpawn` consumes the hint **only when the stage matches** — so a
mid-review restart re-enters review (resuming that session), never a fresh implement, and a recovered
implement session is not misapplied to a review stage.
