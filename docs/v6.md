# Beckett v6 — the teammate generation

> **Status: design contract, not yet live.** [`docs/ARCHITECTURE.md`](ARCHITECTURE.md) remains the
> authoritative doc for the running system (v5.x). This document is the build contract for the v6
> generation, the way [`specs/_legacy-v3/V3.md`](../specs/_legacy-v3/V3.md) was for v3. When v6
> ships, its content folds into ARCHITECTURE.md and this file becomes history.

## The thesis in one paragraph

Every Beckett generation so far has been **invocation-shaped**: a human @mentions Beckett, the
Concierge sizes the effort, and machinery executes. v5 pushed that shape as far as it goes —
concurrent per-channel sessions, ambient interjection, a background browser agent, casting,
multiplayer memory. v6 changes the shape itself. **The unit of input is no longer the request; it
is the event.** A human message is just one event source among many — ticket movement, CI results,
mail, timers, memory maintenance findings, its own idle capacity. Beckett perceives all of it,
decides continuously what deserves attention, forms its own intentions, acts through the same
gated machinery v5 already trusts, and learns from every outcome. Humans stop asking for things
to get done. They talk the way teammates talk, and Beckett places itself — in the conversation
and in the work — the way a sharp colleague does: sometimes with a reply, sometimes with a PR,
most of the time with silence.

## Verdict on the "Coworker as a Service" flow

The candidate integration diagram (Chat Agent → Concierge Agent → Quick/Workers/Review, a
"Bored" idle monitor, a unified memory store, and a Propose→Validate→Apply self-improvement
loop with a rollback gate) gets several things right, and we adopt them:

- **A closed self-improvement loop whose outputs are concrete artifacts** (edited prompts, new
  skills, routing changes) rather than vibes. v5 has the apply path (`self-improve` skill →
  ticket → PR → deploy) but not the loop.
- **An evaluation-and-rollback gate**: only validated changes ship, and there is a last-known-good
  to fall back to. v5's evals (`src/eval/`, `scripts/eval/`) compare *models*; nothing regression-
  tests *Beckett's own behavior* before a deploy. That gap is real and v6 closes it (§ Improve).
- **Async write-behind memory**: recall should never block a turn on writes. Adopted (§ Learn).
- **A fast path for simple work**: exists in v5 as the quick lane; kept.

But as the flow for giving Beckett total agency, the diagram is wrong in four load-bearing ways:

1. **It is still hub-and-spoke around the user.** The entry point at the top of the page is
   `User → Chat Agent`. "Routines" and "Bored" are bolted onto the side as extra inlets into the
   same request pipeline. That is v5 with more plumbing: proactivity remains an *input adapter*
   to an invocation machine. Total agency inverts the diagram — perception is the top of the
   page, and a human message is one perception source with no privileged position in the
   architecture (plenty of privilege in *judgment*: people outrank cron ticks; see § Attend).
2. **The Chat Agent / Concierge Agent split reintroduces the split-brain the memory box claims
   to remove.** v5 already learned this lesson the hard way: the one who talks must be the one
   who decides. Beckett's voice, its effort sizing, and its social judgment live in one
   long-lived agent (`src/concierge/`) precisely so nothing is lost in a hand-off between a
   "face" process and a "brain" process. Splitting them means every nuance a teammate expressed
   in conversation must survive a serialization boundary before it reaches the thing planning
   the work. We keep one Concierge.
3. **"Bored — monitors for opportunities and idle capacity" is a label, not a design.** The box
   is the easy part. The hard parts are: where opportunities come from (a durable goal ledger
   with provenance, § Intend), what bounds initiative so it never becomes spam (attention
   economics and a learned shut-up gate, § Attend), and what licenses the resulting actions
   (the agency gate, § The license). v5's ambient pipeline already proved the binding
   constraint is social calibration, not idle CPU.
4. **It solves self-improvement by hiring three more standing agents** (Efficiency, Learning,
   Improvement — then note 5 admits they should merge). Beckett's v5 lesson: agents are cheap,
   contracts are expensive. Standing meta-agents are three more prompts to drift and three more
   processes to supervise. In v6 the Propose→Validate→Apply loop is not new residents — it is a
   *pipeline over artifacts* that reuses the machinery that already exists and is already
   trusted: proposals are tickets on the beckett project, validation is a gate stage in the
   dispatcher, apply is the existing deploy path. (§ Improve.)

One more gap worth naming: the diagram has no trust model. A "single store for all memory"
without an audience model is a privacy regression against v5, which enforces
public/owner/dm visibility fail-closed in code (`src/memory/`), and total agency without action
classes is how an agent loses its seat at the table. v6 treats the license as a first-class
subsystem, not an afterthought.

## The v6 loop

Six verbs, one cycle, running continuously:

```
        ┌──────────────────────────────────────────────────────────┐
        │                                                          │
   SENSE → ATTEND → INTEND → ACT → LEARN → IMPROVE ────────────────┘
   events   triage   goals    the    lessons  gated changes to
   from     + budget ledger   v5     + memory doctrine, casts,
   every    + shut-up         effort           skills, flows,
   surface  gate              ladder           and the loop itself
```

Each stage below states what v5 already has (with file paths), what v6 adds, and the contract.

### Sense — everything is an event

**v5 has:** Discord messages reach the Concierge only as @mentions/DMs (`src/discord/access.ts`)
or as ambient candidates where enabled (`src/concierge/ambient.ts`); ticket updates arrive as
`SYSTEM` turns; AgentMail is polled; routines fire on humanized schedules (`src/routine/`);
memory runs a nightly `maintain()` pass (`src/memory/maintain.ts`).

**v6 adds:** one **perception bus**. Every surface emits a typed `Percept` into a single
prioritized queue the daemon owns: channel messages (all of them, in ambient-enabled channels —
not just candidates that pre-passed a "reply?" framing), ticket/PR/CI events, inbound mail,
routine fires, telemetry anomalies (crash-loop alarms, spend spikes from `src/spend.ts`),
memory-maintenance findings ("this note contradicts that one"), and a low-frequency **idle tick**
that fires when the fleet has spare capacity. The existing ambient burst-assembly (8s cold /
4s engaged lulls) becomes the Discord adapter's batching policy rather than the whole story.

**Contract:** nothing acts from inside Sense. Adapters normalize and enqueue; all judgment lives
downstream. This is the same discipline as the v5 tracker loop (poller emits, dispatcher decides)
lifted to every surface.

### Attend — the economics of initiative

**v5 has:** a cheap classifier (Haiku/Cerebras, `src/concierge/triage.md`) that answers "may
Beckett reply here?", per-channel proactivity levels (`proactivity-store.ts`: off → mention →
ambient → auto), channel profiles (`channel-profiles.ts`), and a doctrine-level high bar for
"Proactive:" tasks.

**v6 adds:** attention as a budgeted resource. The triage cascade becomes two stages:

- **Relevance** (cheap, per-percept): is there anything here for Beckett at all? Unlike v5
  triage, the output space is not reply/pass — it is **speak / act / learn / pass**. A message
  that needs no reply may still carry a fact worth remembering or a commitment worth tracking.
- **Worth** (Concierge judgment, per-batch): `value × confidence × timeliness` against an
  **initiative budget** — per-channel and per-person caps on unprompted contributions per day,
  spent by speaking or by starting visible work, refunded by positive feedback. Human-origin
  percepts get standing priority over machine-origin ones; a person mid-sentence always
  outranks an idle tick.

**The shut-up gate is a learned artifact, not a constant.** Every unprompted contribution logs
its reception (replies, reactions, silence, an owner "not now") as a signal into the lessons
store (§ Learn). Per-channel norms — this room hates drive-by suggestions, that room expects
them — are recalled at Attend time and move the bar. Silence is a first-class, logged outcome:
"saw it, chose not to," distinguishable from "never saw it," so calibration can be audited.

**Contract:** Attend can never act. It can only route a percept batch to Intend (as a goal
candidate), to the Concierge (as a turn), to memory (as a learn), or to the floor.

### Intend — the goal ledger (the real "Bored" engine)

**v5 has:** nothing. This is the largest genuinely-new subsystem in v6. Today "proactive" means
reacting to ambient conversation; there is no place where Beckett's own open loops live.

**v6 adds:** a durable **goal ledger** — the agentic sibling of the task registry (`src/task/`).
A goal is smaller than a task and bigger than a thought: a thing Beckett believes should happen,
with provenance, a value estimate, a staleness clock, and a state
(`open → planned → executing → done | dropped`). Goals enter the ledger from:

- **conversation extraction** — the Concierge hears "ugh, we should really fix the flaky deploy
  someday" in ambient flow and files a goal *with a link to the message that spawned it*;
- **commitments Beckett made** — "i'll take a look tomorrow" becomes a goal with a due window,
  so Beckett stops being the teammate who says that and never does;
- **its own operations** — a crash-loop alarm, a repeatedly-reworked ticket, a doctor probe
  that keeps warning;
- **memory maintenance** — contradictions and stale facts that need a human to resolve;
- **the improve pipeline** — accepted improvement proposals park here until capacity frees up.

When the idle tick fires, Intend ranks open goals (value × staleness × fit-to-current-capacity)
and promotes the winner into the v5 effort ladder exactly as if a human had asked: inline answer,
quick agent, browser errand, or a numbered task with branches. **Goals are owner-visible**
(`beckett goals`, and a periodic "here's what i'm sitting on" digest on request): total agency
with a public agenda, never a private one.

**Contract:** promotion out of the ledger goes through the same Act machinery and the same
license as human-initiated work. There is no side door where a goal becomes an action without
passing the agency gate.

### Act — the v5 machinery, kept

This is where v6 is deliberately conservative, because v5's execution layer is its proven asset:
the Concierge files work, the tracker (`src/bored/`, `src/tracker/`) is the queue, the
dispatcher (`src/dispatch/dispatcher.ts`) turns state changes into cast worker spawns in
isolated worktrees (`src/worker/worktree.ts`), comments steer live workers, review casts gate
merges, and the quick/browser lanes handle errands. None of that changes shape.

What changes is **flow adaptability**. v5.9 already moved stage definitions into one registry
(`src/dispatch/stages.ts`: implement/review/design/design_check) and made agents data-only
(`src/agent/`). v6 finishes the thought: a ticket's stage sequence becomes a declared **flow** —
a small DAG over registry stages — chosen (or composed) per ticket instead of hard-coded.
The INT board's Design → parked-check → Implement → Review pipeline stops being a special case
and becomes just one flow among several; a hotfix flow might be implement-with-self-review; a
high-stakes flow adds a second-opinion review cast on a different harness. Because flows and
casts are data, **the improve pipeline can ship a new flow without touching the dispatcher** —
that is what "adaptable agentic flows" means concretely.

Coding accuracy rides the same rails: acceptance criteria harden into an executable convention
(a failing-then-passing test is part of the definition of done wherever the repo supports it),
review-tier escalation stays (`reviewTier` self/fresh by effort), and workers get
**lesson-injected context** — before a spawn, the dispatcher recalls lessons keyed by repo,
stage, and harness (§ Learn) and prepends them to the cast prompt, so the fleet stops making
last month's mistakes.

### Learn — from facts to lessons

**v5 has:** a knowledge graph of durable facts about people, projects, and environment
(`src/memory/`), with fail-closed visibility scoping, Moss-accelerated retrieval
(`src/moss-local/`), agentic recall, and a nightly maintenance pass. It remembers *who and
what*. It does not remember *how it went*.

**v6 adds:** an **episodic lessons store** beside the graph. Every consequential outcome
distills into a small structured record: what was attempted, cast, what the reviewer caught,
what a human corrected, how an unprompted contribution was received, what a rollback reverted.
Writes are async write-behind — nothing in a turn or a worker teardown blocks on them.
The nightly `maintain()` pass gains a promotion step: recurring lessons (the same reviewer
catch three times; the same channel cold-shouldering a category of interjection) get promoted
into **proposal candidates** for the improve pipeline. Retrieval stays the existing recall
machinery — lessons are notes with a type, not a new database; visibility scoping applies to
them exactly as to facts.

**Contract:** lessons are evidence, not authority. A lesson can change behavior only by being
recalled into a prompt (soft, immediate) or by being promoted into a validated proposal
(hard, gated). No component silently rewrites its own instructions because a lesson said so.

### Improve — the closed loop, on existing rails

**v5 has:** the apply path only. The `self-improve` skill splits changes into live-editable
(persona, memory) and repo-owned (skills, doctrine, code), and repo-owned changes go through a
`--project beckett` ticket → PR → review → deliberate deploy (`deploy/deploy-prod.sh` with
typecheck-before-restart, health read-back, and version tagging in `src/version/`).

**v6 adds:** the loop around that path, mapping the diagram's Propose→Validate→Apply onto rails
that already exist instead of onto three standing meta-agents:

- **Propose.** Sources: promoted lessons, telemetry (spend, latency, rework counts), eval
  drift, and the owner. A proposal is a ticket on the beckett board with a structured header:
  the change, the evidence links (lessons/metrics that motivated it), and the predicted effect.
  Filing a proposal is FREE-class; it is just a ticket.
- **Validate.** A **gate stage** in the stage registry, run in the worker's worktree before the
  PR advances: `bun x tsc --noEmit`, `bun test`, the triage-classifier eval
  (`scripts/eval/`), and — new — a **golden behavior suite**: a curated set of recorded
  Discord scenarios (effort sizing, shut-up-gate cases, addressee judgment, decline cases)
  replayed against the changed doctrine/prompts with graded expectations, plus golden coding
  tasks for cast/flow changes. The suite lives in-repo so it version-controls with the behavior
  it protects. Fail → the proposal ticket gets the report and parks; nothing ships.
- **Apply.** The existing deploy, unchanged in spirit: deliberate, typechecked, health-checked,
  version-tagged. v6 adds an **observation window** — for a period after a self-originated
  deploy, the daemon compares its live health signals (crash alarms, turn error rate, budget
  burn) against pre-deploy baseline, and a regression triggers an automatic revert to the
  previous tag plus an owner-visible incident note. Last-known-good is real, not aspirational.

**Contract:** the improve pipeline may change prompts, skills, casts, flows, routines, triage
thresholds, and Beckett's own source — but it may never widen its own license. Changes to the
agency gate's action classes, to autonomy levels, or to the validate gate itself are
ALWAYS-ASK: proposed, argued for with evidence, and approved by the owner in Discord. The loop
improves the worker; the owner licenses the citizen.

## The license — autonomy without a leash is not the goal

Total agency does not mean unbounded agency; it means **Beckett initiates, and the license
decides what initiative may touch.** The v5 agency gate (`src/agency/`) — every outward action
classified FREE / HANDSHAKE_GATED / ALWAYS_ASK, fail-closed — is the constitution of v6, not a
speed bump to remove. What v6 changes is *who starts things*, never *what may happen without a
human*.

Two extensions:

- **A `teammate` proactivity level** above `auto` in the per-channel ladder: Beckett may start
  goal-ledger work affecting that channel's projects unprompted, announcing intent as it does
  ("taking the flaky-deploy thing — #57, holler if that's wrong"). The announcement turns the
  handshake ambient: cheap for humans to veto, structural for Beckett to honor. Channels opt
  *in*; the default ships exactly as conservative as v5.
- **The trust ratchet.** Autonomy expansions are themselves proposals through the improve
  pipeline: evidence in, owner approval required, and reversible the same way any deploy is.
  Trust is earned in public, in writing, per capability — the same way a human teammate earns
  scope.

## Groups — where proactive intelligence compounds

The user-visible payoff of v6 is strongest in multi-human channels, and v5 already built the
substrate: channel-scoped shared context, per-channel sessions, addressee judgment, provenance
and visibility-scoped multiplayer memory, and federation for sibling Becketts. v6's group
behavior stands on it:

- **Placement over interjection.** The Attend stage's learned norms decide not just *whether*
  to speak but *where and as what* — a reply in flow, a thread on the side, a goal silently
  filed, a PR that shows up with context. A good teammate's best contributions are often not
  messages.
- **Working out loud.** Goal-ledger work extracted from group conversation is announced in the
  room that spawned it, with the provenance link. Nobody discovers that Beckett has been
  quietly building something; they watch it pick work up the way they watch each other.
- **The room teaches the agent.** Reception signals in group channels are the densest lesson
  source Beckett has — every unprompted contribution is a small experiment in social
  calibration, logged and learned from. This is why groups *promote* proactive intelligence
  rather than merely tolerating it: more people, more feedback, faster calibration.

## Milestones — v6 ships as a ratchet, not a rewrite

Each stage lands separately, deployable and reversible, in dependency order:

- **v6.0 — Perception bus.** Unify all inlets behind typed percepts + the four-way relevance
  triage (speak/act/learn/pass). Behavior parity with v5 by default; the ambient path becomes
  one adapter among several.
- **v6.1 — Lessons.** The episodic store, async write-behind, outcome distillation from
  dispatcher/reviewer/ambient reception, lesson-injected worker context.
- **v6.2 — Goal ledger.** Conversation extraction, commitments, ops-originated goals, the idle
  tick, `beckett goals`, promotion through the effort ladder. `teammate` level ships here,
  opt-in per channel.
- **v6.3 — Validate gate.** The golden behavior suite + gate stage in the stage registry;
  every repo-owned self-change passes through it from here on.
- **v6.4 — Closed improve loop.** Lesson promotion → proposals → validate → deploy with
  observation window and auto-revert. This is the first release where Beckett routinely gets
  better without being asked to.
- **v6.5 — Adaptive flows.** Per-ticket flow DAGs over the stage registry; INT normalizes into
  a flow; the improve loop may ship new flows as data.

Invocation never breaks: `@beckett do X` works identically at every stage. v6 makes invocation
*optional*, not obsolete.

## Non-goals

- **No Chat/Concierge split.** One agent owns voice and judgment, per lesson learned.
- **No standing meta-agents.** Improve is a pipeline over tickets, not new residents.
- **No unified-memory flattening.** Working context, the knowledge graph, and lessons have
  different lifetimes and different audiences; visibility scoping stays fail-closed and
  code-enforced.
- **No self-widening license.** The agency gate and autonomy levels change only by owner
  approval, ever.
- **No Discord voice in v6.** Real but orthogonal; it multiplies presence, not agency. It
  slots cleanly after the perception bus exists (voice is just another adapter) and is
  deferred to keep v6 honest.
