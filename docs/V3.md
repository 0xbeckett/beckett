# Beckett v3 — Plane Ticket-Queue: the build contract

This is the integration backbone. Every parallel build agent reads this before writing code.
The Contract phase has already landed the shared seams (types, config, cast helpers); the rest
of this doc tells you exactly what to build, what to import, and where the boundaries are so two
agents never edit the same surface.

## 0. The architecture in one paragraph

A **Concierge** (a long-lived `claude -p` Opus agent) owns Discord. It chats in Beckett's voice,
decides effort, and for real work FILES A TICKET into Plane (self-hosted at
`plane.0xbeckett.me`) with per-stage **casting**. It never does the work itself. The **shell**
polls the Plane REST API every `config.plane.poll_secs` and emits `PollEvent`s. A **Dispatcher**
consumes those events: a ticket entering `in_progress` spawns `casting.implement`'s harness as a
worker (git worktree, under scope-guard); `in_review` spawns `casting.review`'s harness as a
reviewer; a new comment on an in-flight ticket is injected as a STEERING nudge to the live worker;
`cancelled` aborts the worker; when a worker finishes the dispatcher advances the ticket state and
posts a summary comment.

**Retired (do NOT touch / depend on):** `src/state/orchestrator.ts`, `src/brain/*`. v3 replaces the
v2 control core. The v2 `WorkerManager`/`Store`/DAG types still exist but the v3 dispatcher does
**not** route through `DefaultWorkerManager` (it is bound to the v2 `Store`). See §6.

## 1. Import + style conventions (non-negotiable — match neighbors)

- **Runtime:** bun. **Module system:** ESM. **Language:** TypeScript (`strict`,
  `noUncheckedIndexedAccess`).
- **Relative imports carry an explicit `.ts` extension** (bun-native, `allowImportingTsExtensions`):
  `import { Ticket } from "./types.ts";` — never extensionless, never `.js`.
- **Logger:** `import { log } from "../log.ts";` then `const logger = log.child("plane.client");`
  (or accept an injected `Logger` and default to `log.child(...)`, like `manager.ts`). Never
  `console.log`. JSON-to-stdout is only for the CLI (`src/cli/beckett.ts`).
- **External input is validated with zod** (the cast JSON already is in `cast.ts`; validate Plane
  REST responses the same way at the client boundary).
- **ids:** mint via `src/ids.ts` (`prefixedId("...")`); don't hand-roll ids.
- **No `any`** unless genuinely unavoidable; **no TODOs/stubs** — implement fully.
- **Factory style:** export a `createX(deps)` alongside the class (see `createWorkerManager`).

## 3. PlaneClient — exact method signatures (`src/plane/client.ts`)

`PlaneClient` is the ONLY module that speaks HTTP to Plane. It hydrates raw issues into `Ticket`s
(running `parseCast` on the description) and translates `TicketState` ⇄ Plane workflow state UUIDs
using `config.plane.state_map`. Secret: read `process.env.PLANE_API_TOKEN` (never from config).

```ts
import type { Config, Logger } from "../types.ts";
import type { Ticket, TicketState, PlaneComment, Casting } from "./types.ts";

export interface CreateTicketInput {
  title: string;
  body: string;            // human prose; serializeCast composes the final description
  casting: Casting;
  criteria: string[];
  state?: TicketState;     // default "backlog"
  assignees?: string[];
}

export interface PlaneClientDeps {
  config: Config;
  token?: string;          // defaults to process.env.PLANE_API_TOKEN
  logger?: Logger;
  fetch?: typeof fetch;    // injectable for tests
}

export declare class PlaneClient {
  constructor(deps: PlaneClientDeps);

  /** All issues in the configured project, hydrated to Tickets. `updatedSince` (ISO) narrows. */
  listIssues(opts?: { updatedSince?: string }): Promise<Ticket[]>;

  /** One issue by Plane id, hydrated; null if 404. */
  getIssue(id: string): Promise<Ticket | null>;

  /** Create an issue; serializes casting+criteria+body into the description. Returns hydrated. */
  createIssue(input: CreateTicketInput): Promise<Ticket>;

  /** Move a ticket to a new lifecycle state (resolves state_map name → Plane state UUID). */
  setState(id: string, state: TicketState): Promise<void>;

  /** Comments on an issue, oldest→newest. `since` (ISO) returns only newer comments. */
  listComments(ticketId: string, since?: string): Promise<PlaneComment[]>;

  /** Post a comment (used for worker summaries + dispatcher status). Returns the created comment. */
  addComment(ticketId: string, body: string): Promise<PlaneComment>;
}

export function createPlaneClient(deps: PlaneClientDeps): PlaneClient;
```

Hydration rule (the contract the poller/dispatcher rely on): for every returned `Ticket`,
`{ casting, criteria, body } = parseCast(rawDescription)` and `description = rawDescription`.
`state` is the reverse `state_map` lookup (Plane state name → `TicketState`); an unmapped Plane
state defaults to `"backlog"` and logs a warn.

## 4. Poller — `src/plane/poll.ts`

Holds the last snapshot (`Map<ticketId, { state, updatedAt, lastCommentAt }>`). Each tick calls
`client.listIssues({ updatedSince })`, diffs against the snapshot, and returns `PollEvent[]`:

- ticket id unseen → `{ kind: "created", ticket }`.
- `state` differs → `{ kind: "state_changed", ticket, from, to }`. If `to === "cancelled"`, ALSO
  emit `{ kind: "cancelled", ticket }` (dispatcher uses `cancelled` to abort; `state_changed`
  remains the audit signal).
- newer comments (via `client.listComments(id, since)`) → one `{ kind: "comment_added", ticket,
  comment }` per new comment.

```ts
export interface PlanePollerDeps { client: PlaneClient; logger?: Logger; }
export declare class PlanePoller {
  constructor(deps: PlanePollerDeps);
  /** One poll cycle: fetch, diff, advance the snapshot, return new events (ordered). */
  poll(): Promise<PollEvent[]>;
  /** Seed the snapshot from current Plane state WITHOUT emitting (cold-start / recovery). */
  prime(): Promise<void>;
}
```

The shell drives `poll()` on a `config.plane.poll_secs` interval and hands each event to the
dispatcher. The poller is pure-ish (no spawning) — it only reads Plane and diffs.

## 5. Dispatcher — `src/plane/dispatcher.ts` (the state machine)

Single consumer of `PollEvent`s. Keeps an in-memory map `ticketId → live worker handle` (the
return of §6 `spawnTicketWorker`). State-machine table:

| PollEvent | Condition | Action |
|---|---|---|
| `state_changed` → `in_progress` | no live worker for ticket | spawn worker from `casting.implement` (default harness `claude`); on `finished:success` → `client.setState(id,"in_review")` + summary comment; on `finished:error` → comment + leave in `in_progress` (or `todo`) for human |
| `state_changed` → `in_review` | no live reviewer | spawn reviewer from `casting.review` (default harness `claude`); reviewer verdict comment; pass → `setState(id,"done")`, fail → `setState(id,"in_progress")` (re-work) |
| `comment_added` | live worker exists AND comment author ≠ Beckett | `handle.nudge(comment.body)` (STEERING) |
| `comment_added` | no live worker | ignore (or Concierge handles) |
| `cancelled` | live worker exists | `handle.abort("ticket cancelled")`, reap, drop handle |
| `state_changed` → `done`/other | — | reap any live worker; no spawn |
| `created` | — | no spawn (work starts on `in_progress`); may log |

Defaults: if `casting.implement`/`casting.review` is absent, use `{ harness: "claude" }`. Map
`HarnessSpec.effort` → the worker envelope (reuse the effort→turns/wallclock mapping pattern in
`manager.ts`). Beckett-authored comments must be tagged/recognizable so steering ignores its own
summaries (e.g. prefix or known author id) to avoid a self-nudge loop.

```ts
export interface DispatcherDeps {
  client: PlaneClient;
  config: Config;
  resolveRepoRoot: (ticket: Ticket) => string;   // ticket → absolute git repo root
  logger?: Logger;
}
export declare class Dispatcher {
  constructor(deps: DispatcherDeps);
  handle(event: PollEvent): Promise<void>;        // route one event per the table above
  live(): { ticketId: string; workerId: string }[];
}
```

## 6. Spawn contract — how the dispatcher spawns a worker (`src/plane/worker.ts`)

Do **not** route through `DefaultWorkerManager` (bound to the v2 `Store`/`NodeRecord`). Compose the
lower-level primitives the Contract verified, exactly as `manager.ts#startWorker` does, minus the
persistence:

1. **Driver:** `import { createDriver } from "../drivers/index.ts";` →
   `const driver = createDriver(spec.harness, config, logger);` (claude today, codex once §7 lands).
2. **Worktree:** `import { createWorktree, removeWorktree, excludeFromGit } from "../worker/worktree.ts";`
   `await createWorktree({ repoRoot, workspace, branch, baseRef, reuseIfExists: false });` with
   `workspace = join(repoRoot, ".beckett", "worktrees", workerId)` and
   `branch = "beckett/" + workerId + "/" + ticket.identifier`.
3. **Scope-guard:** write `<workspace>/.claude/settings.json` from
   `renderClaudeSettings([scopeGuardSpec(scopeGuardPath, workspace, ownedGlobs)])`
   (`import { scopeGuardSpec } from "../hooks/scope-guard.ts";`,
   `import { renderClaudeSettings } from "../hooks/registry.ts";`). `scopeGuardPath` =
   `join(import.meta.dir, "../hooks/scope-guard.ts")`. Also `excludeFromGit(workspace,
   [".claude/", ".beckett/"])` and write the done-schema to `.beckett/done-schema.json`.
4. **Spawn:** build a `SpawnSpec` (from `../types.ts`) and `await driver.spawn(spec)`. The brief
   (`spec.prompt`) is `ticket.title` + body; `spec.systemAppend` carries scope + the `criteria`.
5. **Control:** STEERING = `driver.sendNudge(text)`; CANCEL = `driver.abort(reason)`; FINISH =
   subscribe `driver.onEvent(e => …)` and act on `e.kind === "finished"` (`e.status` =
   `"success" | "error"`). Telemetry via `driver.getTelemetry()`.

Expose a single entry point the dispatcher calls:

```ts
export interface TicketWorkerHandle {
  workerId: string;
  ticketId: string;
  stage: string;                                  // "implement" | "review" | ...
  nudge(text: string): Promise<void>;
  abort(reason: string): Promise<void>;
  onFinished(cb: (status: "success" | "error", summary: string) => void): void;
  reap(): Promise<void>;                          // removeWorktree + unsubscribe
}
export function spawnTicketWorker(args: {
  ticket: Ticket;
  stage: string;
  spec: HarnessSpec;
  config: Config;
  repoRoot: string;
  baseRef: string;
  logger?: Logger;
}): Promise<TicketWorkerHandle>;
```

## 7. Codex driver — `src/drivers/codex.ts` (implements the Driver interface)

The codex driver implements the SAME `HarnessDriver` interface from `src/types.ts` that
`ClaudeDriver` implements — that is the exact interface to satisfy:

```ts
export interface HarnessDriver {
  readonly kind: DriverKind;                       // codex → "codex-exec-oneshot"
  spawn(spec: SpawnSpec): Promise<SpawnResult>;    // { sessionId, pid }
  sendNudge(msg: string): Promise<NudgeReceipt>;   // { accepted: "delivered"|"queued", at }
  pause(): Promise<void>;
  resume(): Promise<void>;
  abort(reason: string): Promise<void>;
  onEvent(cb: (e: WorkerEvent) => void): () => void;  // returns unsubscribe
  getTelemetry(): WorkerSpend;
}
```

`kind = "codex-exec-oneshot"` (already in the `DriverKind` union in `types.ts`). codex is
exec-oneshot, so `sendNudge` queues for the next resume and returns `{ accepted: "queued" }`
(claude's mid-turn steer is not available); the `WorkerEvent` parser must be tolerant exactly like
`ClaudeDriver.handleLine` — switch on known shapes, route the rest to `{ kind: "unknown" }`, never
throw. `usdEstimate` in `getTelemetry()` is `null` for codex (per `WorkerSpend` contract).
Auth: subscription only — strip `OPENAI_API_KEY`/`ANTHROPIC_API_KEY` from the child env (mirror
`ClaudeDriver.childEnv` + `FORBIDDEN_ENV_KEYS`). Read `config.harness.codex` for `bin`,
`default_model`, `sandbox_mode`, `approval_policy`, `network_default`.

**Registration (the only `index.ts` edit):** uncomment/add in `FACTORIES`:
```ts
import { CodexDriver } from "./codex.ts";
const FACTORIES = { claude: (c,l)=>new ClaudeDriver(c,l), codex: (c,l)=>new CodexDriver(c,l) };
```

## 8. Concierge — `src/concierge/*` + `beckett ticket` CLI

The Concierge is a long-lived `claude -p` Opus agent (model from `config.concierge.model`,
default `claude-opus-4-9`) that owns Discord I/O via the existing `DiscordGateway`
(`src/discord/gateway.ts`). It chats in voice, classifies effort, and for real work files a ticket
by shelling `beckett ticket create ...`. It NEVER spawns workers itself — that's the dispatcher.

Add a `ticket` command group to `src/cli/beckett.ts` (append a new `if (group === "ticket")` block;
do not touch existing groups). It uses `PlaneClient` in-process (like the `memory` group uses
`createMemory`). Suggested subcommands (JSON to stdout, the Concierge reads it):

- `beckett ticket create --title <t> [--body <b>|--body-stdin] [--state backlog] [--cast <json>] [--criteria a;b;c]`
  → `client.createIssue({...})`, prints `{ id, identifier, url, state }`.
- `beckett ticket comment <id> --body <b>|--body-stdin` → `client.addComment`.
- `beckett ticket state <id> <ticketState>` → `client.setState`.
- `beckett ticket get <id>` / `beckett ticket list [--state s]` → hydrated Ticket(s).

`--cast` accepts the casting JSON (the same object `parseCastJson` validates). Compose the
description with `serializeCast(casting, criteria, body)` — or pass body/casting/criteria to
`createIssue`, which serializes internally. Keep the cast/criteria contract identical to §0.

## 9. Cast block format (canonical — reproduced from `cast.ts`)

Stored INSIDE the Plane issue description:

````
<human prose body…>

```beckett-cast
{ "implement": {"harness":"codex"},
  "review": {"harness":"claude","model":"claude-opus-4-9"} }
```

## Acceptance criteria
- first criterion
- second criterion
````

- Fence tag is exactly `beckett-cast`; JSON is a stage-name → `HarnessSpec` map.
- Criteria heading is exactly `## Acceptance criteria`; bullets are `- ` / `* ` lines until the
  next `## ` heading or end.
- `parseCast(description) → { casting, criteria, body }`; `serializeCast(casting, criteria, body) →
  description`. They round-trip (verified). A missing/malformed cast block → `casting = {}`; a
  missing criteria section → `criteria = []`. Never throws.

## 10. Config surface (landed by Contract)

`config.plane`: `base_url`, `workspace_slug`, `project_slug`, `poll_secs` (default 15),
`state_map.{backlog,todo,in_progress,in_review,done,cancelled}` → Plane state names (defaults
`Backlog`/`Todo`/`In Progress`/`In Review`/`Done`/`Cancelled`). `config.concierge.model` (default
`claude-opus-4-9`). The secret `PLANE_API_TOKEN` is read from `process.env` (the `.env` loader in
`config.ts` deliberately allows it through; only `ANTHROPIC_API_KEY`/`OPENAI_API_KEY` are dropped).
